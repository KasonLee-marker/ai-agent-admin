## 1. 概述

### 1.1 背景

当前语义切分策略在上传文档时会同步调用 Embedding API，导致上传接口响应慢（可能需要几秒甚至更长）。用户体验不佳。

### 1.2 目标

- 上传接口响应时间 < 1秒（语义切分时）
- 用户能在文档列表看到处理进度
- 处理失败时用户能感知并重试

### 1.3 范围

**包含：**

- 语义切分改为异步处理
- 文档状态新增"处理中"
- 进度显示（当前句子数/总句子数）

**不包含：**

- 其他分块策略（FIXED_SIZE、PARAGRAPH、SENTENCE、RECURSIVE 保持原有流程）
- 进度条 UI（先做简单数字显示）

## 2. 需求分析

### 2.1 用户场景

| 角色 | 场景         | 期望结果                |
|----|------------|---------------------|
| 用户 | 上传文档选择语义切分 | 快速返回，看到"处理中"状态      |
| 用户 | 查看文档列表     | 看到处理进度（如 50/100 句子） |
| 用户 | 处理完成后      | 状态变为"已分块"，可点击向量化    |

### 2.2 输入输出

**上传接口变更：**

| 输入               | 类型     | 必填 | 说明               |
|------------------|--------|----|------------------|
| chunkStrategy    | String | 是  | SEMANTIC 时触发异步处理 |
| embeddingModelId | String | 是  | 语义切分必填           |

| 输出               | 类型     | 说明                       |
|------------------|--------|--------------------------|
| status           | String | PROCESSING（语义切分）         |
| semanticProgress | Object | {current, total, status} |

**状态新增：**

| 状态                  | 说明      | 适用策略       |
|---------------------|---------|------------|
| SEMANTIC_PROCESSING | 语义切分处理中 | 仅 SEMANTIC |

### 2.3 约束条件

- Embedding API 每批最多 10 个句子
- 进度更新需要持久化到数据库
- 处理过程需要可中断/可恢复（暂不实现）

## 3. 设计方案

### 3.1 整体架构

```
上传文档 (SEMANTIC) 
    → 创建文档记录 (status=SEMANTIC_PROCESSING)
    → 返回响应（快速）
    → 异步线程开始处理：
        1. 提取文本
        2. 分批调用 Embedding API
        3. 每批完成后更新进度到数据库
        4. 完成后更新状态为 CHUNKED
```

### 3.2 数据模型变更

**Document 实体新增字段：**

```java
// 语义切分进度
@Column(name = "semantic_progress_current")
private Integer semanticProgressCurrent;  // 已处理句子数

@Column(name = "semantic_progress_total")
private Integer semanticProgressTotal;     // 总句子数

// 状态枚举新增
SEMANTIC_PROCESSING  // 语义切分处理中
```

### 3.3 API 设计

**获取进度接口：**

```
GET /api/v1/documents/{id}/semantic-progress

Response:
{
  "code": 200,
  "data": {
    "status": "SEMANTIC_PROCESSING",
    "current": 50,
    "total": 100,
    "percentage": 50
  }
}
```

### 3.4 核心流程

```
1. uploadDocument (SEMANTIC)
   - 创建 Document，status = SEMANTIC_PROCESSING
   - 返回响应（不等待处理）

2. processDocumentAsync
   - 提取文本
   - 按句子分割，计算 total
   - 分批调用 embedding（batchSize=10）
   - 每批完成后：
     - current += batch.size()
     - 更新 Document（semanticProgressCurrent, semanticProgressTotal）
   - 全部完成后：status = CHUNKED

3. 前端轮询进度
   - 每 2 秒调用 /semantic-progress
   - 显示进度：current/total
```

### 3.5 技术选型

| 组件   | 选型            | 原因       |
|------|---------------|----------|
| 异步执行 | Spring @Async | 已有基础设施   |
| 进度存储 | Document 表    | 简单，无需额外表 |
| 前端轮询 | setInterval   | 简单实现     |

## 4. 交互设计

### 4.1 页面结构

**文档列表状态显示：**

| 状态                  | 显示                  |
|---------------------|---------------------|
| SEMANTIC_PROCESSING | "语义切分中 (50/100 句子)" |
| CHUNKED             | "已分块"               |
| COMPLETED           | "已完成"               |

### 4.2 用户操作流程

1. 上传文档 → 选择语义切分 → 选择 Embedding 模型 → 点击上传
2. 快速返回 → 文档列表显示"语义切分中"
3. 进度自动更新（前端轮询）
4. 完成后显示"已分块" → 可点击"开始向量化"

### 4.3 异常处理

| 异常场景             | 处理方式        | 用户提示          |
|------------------|-------------|---------------|
| Embedding API 失败 | 状态设为 FAILED | "语义切分失败: xxx" |
| 用户删除文档           | 取消异步任务      | 直接删除          |
| 模型不可用            | 提前检查        | 上传时就报错        |
